NotiQ — Distributed Notification Platform

Eight Rust microservices communicating over gRPC and a Postgres-native job queue. Built to explore what actually breaks when you operate a reliable delivery system at scale — gossip-based membership, consistent hashing, lock-free concurrency, and end-to-end backpressure across service boundaries.

how it was built — key decisions
01
Postgres as the queue broker
SKIP LOCKED · LISTEN/NOTIFY · outbox · event sourcing
Why Postgres, and where it breaks

A Postgres-native queue (same approach as Graphile Worker, River, Oban) keeps job state transactional with the rest of the application. No external broker, no dual-write race, no new failure domain to operate. The honest tradeoffs: Postgres becomes the throughput ceiling at very high write rates, WAL amplification increases under heavy job churn, and cross-region queue replication is not straightforward. For this project's scale and operational profile, those tradeoffs are acceptable.

SKIP LOCKED for concurrent dequeue

Multiple workers claim different jobs from the same table with zero serialization conflicts — no advisory locks, no SELECT FOR UPDATE blocking. Rows locked by another transaction are skipped entirely. This is the same mechanism used by pgqueue, Graphile Worker, and River. Building it from scratch means understanding exactly what SQS abstracts away.

LISTEN/NOTIFY for low-latency wake

Workers sleep on a Postgres channel and wake on INSERT rather than polling on a fixed interval. Push for low latency, pull for correctness — the outbox CDC task guarantees at-least-once delivery regardless of whether the notification was received. LISTEN/NOTIFY is used only to minimize wake latency, not as the delivery mechanism.

Event log, not status column

Job state is an append-only sequence: Enqueued → Claimed → Failed → Retried → Delivered. Current state is a projection over that log. Audit trail is free. Replay is exact. The same pattern as CockroachDB's range state machine — overwriting a status column destroys history and makes debugging retried failures guesswork.

02
Distributed worker fleet
Consistent hashing · SWIM gossip · lock-free concurrency
Consistent hashing for shard routing

BTreeMap vnode ring over a 2³² token space. 150 virtual nodes per worker smooth load distribution. On join or leave, only the adjacent token range migrates — ~1/N of total keys. Proven with a test: add a worker, measure fraction of keys that moved.

SWIM-lite for membership

Direct ping → suspect on timeout → indirect ping via K random peers → dead on no ack. Epidemic broadcast piggybacks membership deltas on outgoing pings — O(log N) convergence without a central coordinator. A dead event triggers ring rebalance automatically.

Lock-free MPSC between tasks

crossbeam-channel connects the gossip task, dequeue task, and fan-out task inside each worker instance. No mutex on the hot path. AtomicU8 for node liveness state with Acquire/Release ordering — not SeqCst, which would be correct but unnecessarily expensive.

Zero-copy payload fan-out

Notification payloads are wrapped in bytes::Bytes. Fan-out to N delivery channels clones the Arc pointer, not the heap allocation. A criterion benchmark with allocation counting confirms zero extra heap allocations per channel in the fan-out path.

03
Reliability under failure
Bulkhead · circuit breaker · backpressure · chaos testing
Three delivery services, not one

Email, SMS, and webhook are separate ECS services with separate deployment pipelines and failure domains. A SES outage triggering circuit trips has zero effect on Twilio throughput. The bulkhead makes failure boundaries explicit rather than accidental.

End-to-end backpressure

Each delivery service signals its capacity upstream when it's near saturation. worker-svc aggregates these signals via an atomic queue depth counter and propagates the signal upstream to enqueue-svc. Unbounded queues cause silent data loss; this prevents it.

Decorrelated jitter, not exponential

Naive exponential backoff synchronizes retries across instances and causes thundering herd after mass failure. Decorrelated jitter (sleep = min(cap, random(base, prev × 3))) keeps retry waves spread over time. The AWS whitepaper on this is the reference.

Chaos test suite

A --chaos flag randomly kills workers mid-flight, delays acks, and injects network partitions. A test run audits the delivery_log and proves zero message loss. This is the only real proof of reliability — static analysis and code review are not enough.

04
Cloud infrastructure
ECS · RDS · Terraform · IAM · observability
Everything in Terraform

VPC with public/private subnet split, SGs as code, one ECS task definition per service, RDS multi-AZ + read replica, ElastiCache cluster, Secrets Manager, KMS, Route 53, ACM. Nothing clicked in the console — infra is reproducible and reviewable.

IRSA over static credentials

Each ECS task assumes its own least-privilege IAM role via IRSA. enqueue-svc can write to S3 and read from Secrets Manager. delivery-email can call SES. No shared credentials, no static access keys in environment variables or images.

Custom scaling metric

ECS auto-scaling for worker-svc is driven by a custom CloudWatch metric — Postgres job queue depth — not CPU. CPU is the wrong signal for a queue worker: it's low when idle and spiking when overwhelmed. Queue depth is what actually needs to scale against.

OpenTelemetry throughout

Every service emits traces via tracing-opentelemetry. One trace ID spans gateway → enqueue → worker → delivery across gRPC boundaries via propagated context metadata. OTLP export to CloudWatch in prod, Jaeger locally.

05
Service communication choices
gRPC · protobuf · per-service schema · UUID v7
gRPC over REST for inter-service

REST between internal services means parsing JSON, no schema enforcement, and no generated client code. gRPC gives strongly-typed contracts via protobuf, generated client stubs in every service, binary encoding (~3-10× smaller than JSON), and built-in streaming. Breaking changes are caught at compile time, not at runtime in production.

Per-service Postgres schema

Each service owns its tables under a named schema (enqueue, worker, admin). No service queries another's tables directly — all cross-service reads go through gRPC. This enforces bounded context ownership at the database level and makes schema migrations per-service rather than coordinated across the system.

UUID v7 for sortable IDs

UUID v4 is random — index locality is poor, leading to page splits and fragmented B-tree pages under write load. UUID v7 is time-ordered: new rows land at the end of the index, giving sequential write patterns and far better insert throughput on the jobs and job_events tables.

Tokio over async-std

Tokio has the strongest ecosystem alignment for this project: axum, tonic, sqlx, fred, and crossbeam all integrate directly with Tokio's runtime. async-std has a different executor model that creates friction with Tokio-dependent libraries. Ecosystem coherence matters more than runtime benchmarks at this scale.

engineering depth
microservices
Outbox pattern
enqueue-svc writes job + outbox row in one transaction. A CDC task reads the outbox and publishes to worker-svc, then deletes the processed row. This decouples reliable message publishing from the write path without a distributed transaction or dual-write race.
microservices
Saga — choreography vs orchestration
Delivery fan-out uses choreography: each step emits an event, compensation triggers on downstream failure. Multi-step notification flows in scheduler-svc use orchestration: an explicit state machine drives steps and runs compensation in reverse on failure. Both patterns implemented, both understood.
microservices
Custom circuit breaker as Tower::Layer
Gateway-svc wraps downstream gRPC calls in a circuit breaker implemented from scratch as a Tower::Layer + Tower::Service. Closed → Open on error rate threshold, Open → Half-open after timeout, Half-open → Closed on probe success. Composable with other Tower middleware.
microservices
CQRS — separate read and write models
Write commands flow through enqueue-svc to the Postgres primary. Read queries flow through admin-svc against the RDS read replica. Each model is optimized for its access pattern. The write model is normalized for transactional correctness; the read model is denormalized for delivery analytics queries.
microservices
Bulkhead isolation
Three delivery services — email, SMS, webhook — each with its own ECS task definition, deployment pipeline, connection pool, and retry budget. A SES quota breach or Twilio outage is confined to its service. The failure surface is bounded by design, not by hope.
microservices
mTLS inter-service auth
All gRPC calls between services use mutual TLS. Certificates provisioned via ACM Private CA in Terraform. A compromised delivery service cannot impersonate gateway-svc or enqueue-svc. Trust is cryptographic, not network-topology-based.
microservices
Distributed tracing across boundaries
OTel trace context propagated via gRPC metadata headers. A single trace ID spans gateway → enqueue → worker → delivery. When a notification fails, the full causal chain is visible in one trace without manually correlating log timestamps across services.
distributed
SWIM failure detection
Direct ping → timeout → indirect ping via K random peers → suspect state → dead after configurable interval. Epidemic dissemination of membership deltas piggybacks on outgoing pings rather than broadcasting separately. Full cluster converges in O(log N) rounds. False positive rate is tunable against detection latency — a real tradeoff with real consequences.
distributed
Consistent hashing with virtual nodes
BTreeMap over a 2³² token space. 150 virtual nodes per physical worker ensures even distribution even with few instances. A node leaving migrates only the keys between it and its predecessor — ~1/N of total keys. Replication factor 2: every notification is owned by a primary shard and one replica for failure tolerance.
distributed
At-least-once delivery with idempotency
Jobs are retried on failure. A Bloom filter pre-screens idempotency keys (zero false negatives, acceptable false positive rate) before a Redis SET NX confirms deduplication. The two-tier design means ~80% of duplicate checks never reach Redis. Delivery is guaranteed; double-firing is prevented.
distributed
Partition-ordered delivery
Notifications carry a monotonically increasing sequence number scoped to (tenant_id, recipient_id). SKIP LOCKED on the sequence column ensures a recipient's notifications are always claimed in order. The same guarantee as Kafka partition ordering — implemented on Postgres.
distributed
End-to-end backpressure
Delivery services signal saturation upstream to worker-svc via a backpressure flag in gRPC responses. worker-svc tracks this with an atomic queue depth counter and signals enqueue-svc to reject new jobs when depth exceeds threshold. Unbounded queues are the most common silent failure mode in distributed systems.
distributed
Distributed lock via Postgres advisory
scheduler-svc acquires a Postgres advisory lock before firing a cron job. Only one instance fires regardless of replica count. Cheaper and more correct than Redlock for single-database deployments — and the lock is visible in pg_locks for debugging.
system design
Push + pull hybrid delivery
LISTEN/NOTIFY wakes workers with microsecond latency when a job is inserted. SKIP LOCKED handles the actual claim. Push gives low latency; pull gives correctness under failure. Using either alone loses one of those properties — using both keeps both.
system design
Atomic rate limiting in Redis
Token bucket and sliding window rate limiters are implemented as Lua scripts executed atomically in Redis. Two separate INCR + EXPIRE commands would introduce a race — any limit that can be violated under concurrent load is not a limit. Lua atomicity closes that window.
system design
Write fan-out with independent retry
One notification produces N channel jobs — one per delivery service. Each channel has its own retry budget, dead-letter queue, and backpressure signal. A webhook timing out doesn't affect email delivery state. Fan-out makes failure scopes independent.
system design
Multi-tenant isolation at row level
Every table carries a tenant_id column. Per-tenant rate limits, queue depth budgets, and backpressure thresholds are configurable. One noisy tenant sending burst traffic cannot exhaust the delivery capacity of another. Isolation is enforced in the data model, not just the API layer.
system design
Graceful degradation by design
Redis unavailability → rate limiter falls back to allow-all (configurable). SES outage → circuit trips, jobs accumulate in Postgres, delivery resumes on recovery. No single component failure causes data loss. The system degrades predictably rather than failing catastrophically.
system design
Payload tiering to object storage
Notification payloads above a size threshold are written to S3 and referenced by key in the jobs table. Postgres stays fast for metadata queries; S3 handles bulk storage. The same pattern as Kafka's large message support — keep the hot path lean.
Rust
Custom Tower middleware
Circuit breaker implemented as a Tower::Layer + Tower::Service pair. Each state transition (Closed → Open → Half-open → Closed) is modeled explicitly. Composable with other layers in the middleware stack — auth, tracing, and circuit breaking chain without coupling. This is how production Axum and tonic middleware is structured.
Rust
Compile-time SQL validation
sqlx::query! macros check every SQL query against the live database schema at compile time. Column types, nullability, and query results are type-safe. Schema drift that would cause a runtime panic instead causes a compile error — caught before deployment.
Rust
Async stream composition with pin-project
The CDC outbox reader is implemented as a self-referential async stream using pin-project. tokio::select! multiplexes the gossip probe loop and the dequeue poll loop inside each worker task. Waker propagation is explicit — no implicit polling, no missed wakeups.
Rust
Atomic memory ordering
AtomicU8 encodes node liveness (0=alive, 1=suspect, 2=dead). The gossip task writes with Release; the routing task reads with Acquire. SeqCst would also be correct but imposes a global memory fence — unnecessary here because state transitions are monotonic and the two roles are always the same writer/reader pair.
Rust
Graceful shutdown with CancellationToken
A tokio-util CancellationToken propagates through the task tree. Every task selects on the cancellation signal alongside its work loop. On SIGTERM, in-flight jobs drain before the process exits. Proven correct by running the chaos test suite across a rolling ECS deployment.
Rust
Workspace with shared proto library
Single Cargo workspace with 8 service crates and a notiq-common library. tonic + prost generate gRPC client/server stubs from shared .proto files at build time. All services share the same error types, tracing setup, and config loader — no drift between service implementations.
performance
Lock-free MPSC on the hot path
crossbeam-channel replaces a Mutex<VecDeque> between the gossip, dequeue, and fan-out tasks inside each worker. The channel uses a Michael-Scott queue internally — CAS-based, no lock acquisition. Benchmarked under 10k rps: the lock-free path shows measurably lower p99 latency and no lock contention spikes.
performance
Zero-copy payload fan-out
Notification payloads are held as bytes::Bytes — a reference-counted heap slice. Fanning out to 3 delivery channels clones the Arc pointer three times. No heap allocation per channel. A criterion benchmark with allocation counting confirms this: allocation count in the fan-out path is flat regardless of channel count.
performance
Bloom filter to reduce Redis round-trips
A per-process Bloom filter pre-screens idempotency keys. A key that hasn't been seen is guaranteed to pass (zero false negatives). A key that has been seen probably fails (acceptable false positive rate). In duplicate-heavy workloads, ~80% of Redis SET NX calls are skipped entirely.
performance
tokio-console for async profiling
tokio-console instruments the runtime's task scheduler and surfaces task wakeup latency, poll duration, and starvation. Used to find and fix a slow webhook delivery task that was starving the dequeue loop — isolated by spawning deliveries into their own detached tasks.
performance
Atomic queue depth without contention
Queue depth is tracked as an Arc<AtomicUsize> incremented on enqueue and decremented on completion. The backpressure threshold check is a single atomic load. Compared to a Mutex-protected counter, this never blocks on the hot path — critical when enqueue-svc is under burst load.
performance
Connection pool sizing per service
Each service has its own sqlx PgPool with a max size tuned to its query pattern. enqueue-svc runs short write transactions and needs high concurrency. admin-svc runs long analytics queries against the read replica and needs fewer, longer-lived connections. One pool size does not fit all.
algorithms
BTreeMap as consistent hash ring
BTreeMap<u32, WorkerId> maps token positions to worker instances. A clockwise successor lookup is a range() call — O(log N). 150 virtual nodes per worker are inserted at startup by hashing the worker ID with different seeds. The BTreeMap's ordered iteration makes successor lookup trivial without a custom data structure.
algorithms
Min-heap for deadline scheduling
std::collections::BinaryHeap with Reverse<Instant> holds pending jobs ordered by next_run_at. O(1) peek at the nearest deadline, O(log N) insert. The scheduler sleeps until the top entry's deadline rather than polling on a fixed interval — precise wakeup with no busy-waiting.
algorithms
Bloom filter for probabilistic dedup
A Bloom filter gives constant-time membership tests with zero false negatives. An idempotency key that has never been seen is guaranteed to pass. A key that has been seen will probably fail — the false positive rate is a tunable tradeoff against memory usage. Correct use of probabilistic data structures requires understanding this asymmetry.
algorithms
Decorrelated jitter backoff
sleep = min(cap, random_between(base, previous_sleep × 3)). Unlike full jitter, this breaks temporal correlation between retrying instances — consecutive sleeps are randomized relative to each other, not just relative to the base. This is the AWS-recommended algorithm for distributed retry and requires understanding why naive exponential backoff causes thundering herd.
algorithms
Michael-Scott lock-free queue
crossbeam-channel's internal queue is a Michael-Scott queue — a CAS-based MPSC structure with O(1) enqueue and dequeue. Understanding the ABA problem (why pointer tagging is necessary), the CAS loop structure, and why this is safe without a mutex is the depth behind using the library.
algorithms
Sliding window vs token bucket
Token bucket allows controlled burst up to the bucket size then refills at a steady rate. Sliding window counts requests within a rolling time interval with no burst allowance beyond the window limit. Both are implemented as atomic Lua scripts in Redis. The choice depends on whether burst traffic should be tolerated or hard-capped — a real product decision.

decisions worth discussing
storage
SKIP LOCKED gives lock-free row exclusion without advisory locks or serializable transactions. Multiple workers claim different jobs concurrently with no conflicts. Compared to SELECT FOR UPDATE, it never blocks — it simply skips locked rows and moves on.
storage
LISTEN/NOTIFY is not reliable for at-least-once delivery — a worker that crashes between notification and claim loses the message. The outbox CDC task handles delivery guarantees independently. LISTEN/NOTIFY is used only for low-latency wake, not as the delivery mechanism.
storage
Append-only event log instead of a mutable status column. Replaying job_events reconstructs any job's history at any point in time. This makes debugging failures trivial and makes the dead-letter replay in admin-svc exact rather than approximate.
microservices
Outbox pattern over dual-write: writing to the queue atomically with the job record means there's no window where the job exists but hasn't been dispatched — or was dispatched but the write failed. The CDC task provides the eventual publishing guarantee.
microservices
Circuit breaker as a Tower::Layer rather than a global flag: it's scoped to the downstream service it protects, composable with other middleware, and testable in isolation. A global flag would require coordination across all goroutines and can't be per-route.
microservices
CQRS because the write path and read path have genuinely different access patterns. The write path needs low-latency transactional inserts. The admin read path runs multi-join analytics queries over millions of delivery_log rows. Forcing both through the same model means optimizing for neither.
distributed
SWIM over a central heartbeat: a heartbeat server is a single point of failure and a bottleneck at scale. SWIM scales O(log N) and has no coordinator. The tradeoff is that false positive detection latency is tunable but never zero — this is understood and accounted for in the chaos tests.
distributed
Consistent hashing with 150 virtual nodes: without virtual nodes, removing a single physical worker reassigns all its keys to one successor, causing a hotspot. Virtual nodes spread the impact across the ring. 150 per worker is the Cassandra default — a well-studied heuristic, not arbitrary.
distributed
Decorrelated jitter over exponential backoff: after a mass failure, naive exponential backoff synchronizes retries across instances at the same intervals. Decorrelated jitter breaks that synchronization. The thundering herd problem it prevents is not theoretical — it has taken down production systems.
infrastructure
Queue depth as the ECS scaling metric instead of CPU: a queue worker's CPU is low when it's idle and spikes when overwhelmed. By the time CPU triggers scale-out, the queue has already grown. Queue depth is a leading indicator; CPU is a lagging one.
infrastructure
IRSA over a shared IAM role: each service needs exactly the permissions its function requires. enqueue-svc needs S3 write and Secrets Manager read. delivery-email needs SES send. A shared role is a security smell — credential scope should be minimal and auditable per service.
Rust
Acquire/Release on AtomicU8 node state rather than SeqCst: SeqCst imposes a total global order on all atomic operations across all threads — unnecessarily expensive here. The gossip task always writes; the routing task always reads. Acquire/Release is sufficient and correct for this producer/consumer pair.
Rust
bytes::Bytes for zero-copy fan-out: cloning an Arc pointer is two atomic increments. Cloning a heap Vec is a malloc and a memcpy. At 3 delivery channels per notification, zero-copy fan-out eliminates 3 allocations per notification on the hot path. A criterion benchmark with allocation counting confirms this — the count is flat regardless of channel count.
